DNS

Sistema que resuelve nombres (dominios) a IPs y también permite resolución inversa.

Qué tienes que clavar

  • Traduce nombres ↔ IP (resolución directa e inversa).
  • Jerarquía: raíz (.) → TLD → dominio → subdominios.
  • Consulta recursiva: el resolutor te devuelve respuesta final; iterativa: te va redirigiendo.
  • FQDN = nombre completo (conceptualmente termina en un punto que representa la raíz).
  • Servidores raíz: 13 entidades lógicas; muchas instancias distribuidas con Anycast (esto aparece en tu test).
  • MASTER/primario: autoridad de la zona; ahí se notifican los cambios.

Comandos y cosas típicas

  • nslookup dominio: consulta DNS básica.
  • dig/host (si lo usáis): más detalle o alternativa rápida.
  • Comprobar DNS local: /etc/resolv.conf o ajustes de red (según sistema).

Pregunta de desarrollo (del documento)

Enunciado: Explica brevemente qué función cumple el protocolo DNS y describe la diferencia principal entre una consulta recursiva y una consulta iterativa.
Idea-respuesta: El DNS actúa como un traductor que convierte nombres de dominio como google.es en direcciones IP como 216.58.201.163 para que podamos acceder a los servicios de Internet. Las diferencias entre consulta recursiva y iterativa es que en la recursiva el servidor DNS local está obligado a resolver la petición y entregar una respuesta definitiva al cliente preguntando a otros servidores si es necesario mientras que la iterativa devuelve la respuesta solo si la conoce y si no la sabe manda al cliente a otro servidor

Docker

Contenedores = ejecutar apps en entornos aislados y portables, sin cargar un SO completo como una VM.

Ideas clave

  • Contenedor comparte kernel (ligero) vs VM con SO completo (pesado).
  • Imagen = plantilla; contenedor = instancia en ejecución.
  • Dockerfile construye imágenes; Compose orquesta varios servicios con YAML.
  • Volúmenes/bind mounts: persistencia (BBDD y datos).
  • Redes: DNS interno por nombre de servicio en Compose.
  • Ventaja típica: portabilidad y consistencia de entornos.

Comandos que suelen caer

  • docker ps / ps -a: contenedores.
  • docker images: imágenes.
  • docker run: crear/arrancar.
  • docker logs -f: logs.
  • docker exec -it: entrar.
  • docker compose up -d / down: levantar/bajar.

Comparación rápida (lo que suelen preguntar)

  • VM: SO completo (pesado), arranque lento, aislamiento muy alto.
  • Contenedor: comparte kernel (ligero), arranca en segundos, gran portabilidad.
  • Docker Compose: varios servicios (API + BBDD + proxy) definidos en un YAML.
Ejemplo completo: docker-compose.yml (API Node + Mongo con volumen) 
version: "3.9"
services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      MONGO_URI: mongodb://mongo:27017/miapp
    depends_on:
      - mongo
    networks:
      - redapp

  mongo:
    image: mongo:7
    restart: unless-stopped
    volumes:
      - datos_mongo:/data/db
    networks:
      - redapp

volumes:
  datos_mongo:

networks:
  redapp:

Pregunta de desarrollo (del documento)

Enunciado: Explica qué es Docker, por qué surgió y menciona dos ventajas de su uso
Idea-respuesta: Docker es una herramienta que sirve para ejecutar aplicaciones dentro de contenedores, que son entornos aislados donde va incluida la aplicación y todo lo que necesita para funcionar. Docker surgió para evitar problemas cuando una aplicación funciona en un ordenador pero no en otro por diferencias de configuración. Gracias a Docker, las aplicaciones se pueden mover y ejecutar fácilmente en cualquier equipo. Algunas de sus ventajas son que ocupa pocos recursos y que permite arrancar aplicaciones de forma rápida y sencilla.

LDAP

Servicio de directorio para centralizar usuarios/grupos y autenticación; estructura en árbol (DIT).

Lo importante

  • LDAP es directorio (no BBDD relacional): optimizado para lecturas/búsquedas.
  • Estructura en árbol (DIT): base DN → OU → entradas.
  • Entrada = atributos (cn, sn, mail, uid...).
  • DN = ruta única completa dentro del árbol.
  • Autenticación típica: BIND (si conecta, contraseña válida).
  • OpenLDAP: servidor muy usado; herramientas visuales ayudan con el árbol.

Herramientas / comandos típicos

  • Apache Directory Studio: gestión visual.
  • ldapsearch / LDIF (si lo dais): buscar/importar/exportar.
  • Puertos típicos: 389 (LDAP), 636 (LDAPS) si os lo han explicado.

Diferencia LDAP vs MySQL (idea corta)

  • LDAP: árbol jerárquico, lecturas y búsquedas muy rápidas, ideal para autenticación/usuarios.
  • MySQL (relacional): tablas, transacciones y escrituras constantes, ideal para datos de negocio (pedidos, facturas...).

Pregunta de desarrollo (del documento)

Enunciado: ¿Cuáles son las diferencias principales entre una base de datos normal (como MySQL) y un directorio LDAP?
Idea-respuesta: . LDAP organiza los datos en forma de árbol jerárquico (con ramas y hojas) y las Bases de Datos Relacionales organizan los datos en tablas planas (filas y columnas). En cuanto a velocidad LDAP está diseñado para ser extremadamente rápido en lecturas (verificar usuarios muchas veces) y Las Bases de Datos Relacionales son mejores para escrituras constantes y transacciones complejas.

FTP · SFTP · SCP · NGINX

Transferencia de archivos + servidor/reverse proxy. Ojo con diferencias de seguridad y puertos.

Conceptos que suelen caer

  • FTP transfiere archivos pero es inseguro si va en plano.
  • Activo vs pasivo: cambia quién abre el canal de datos y la negociación de puertos.
  • FTPS = FTP + TLS; SFTP/SCP van sobre SSH (lo habitual hoy).
  • SFTP no es 'FTP cifrado': es un protocolo distinto sobre SSH.
  • Cuotas y tipos de usuario: control de espacio y permisos.
  • NGINX: servidor web/reverse proxy para exponer servicios y servir estáticos.

Comandos/config que te salva

  • sftp usuario@host (SSH 22).
  • scp archivo usuario@host:/ruta.
  • nginx -t y systemctl reload nginx.
  • Logs: /var/log/nginx/access.log y error.log.
Ejemplo completo: NGINX como reverse proxy hacia una API 
server {
  listen 80;
  server_name ejemplo.local;

  location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}

Apache Tomcat

Servidor de aplicaciones Java (contenedor de Servlets/JSP). Muy típico: carpetas, WAR y puertos.

Puntos clave

  • Tomcat = contenedor de Servlets/JSP (apps Java web).
  • Servidor web (estático) vs contenedor de servlets (dinámico Java).
  • JSP se compila a servlet; servlet maneja HTTP.
  • Despliegue: .war en webapps.
  • Carpetas típicas: bin, conf, lib, logs, webapps, work, temp.
  • server.xml: conectores/puertos.

Qué repasar rápido

  • Puerto típico: 8080.
  • startup/shutdown (bin).
  • catalina.out / logs/.
Mini-esquema de un .war (por qué preguntan /WEB-INF) 
/
├─ index.jsp (o HTML)
├─ assets/ (css/js/img)
├─ WEB-INF/   (privado, no accesible desde navegador)
│  ├─ web.xml
│  ├─ classes/ (clases .class)
│  └─ lib/     (JARs)
└─ META-INF/ (manifest y metadatos)

Node.js + MongoDB

Backend en JavaScript + base de datos documental. Lo típico: asíncrono, Express, CRUD y documentos/colecciones.

Node.js: lo importante

  • Node.js = runtime de JS en servidor (V8), asíncrono y por eventos.
  • Event Loop: I/O no bloqueante.
  • npm: paquetes; package.json.
  • Express: rutas/middlewares para API REST.

MongoDB: lo importante

  • MongoDB: NoSQL documental (colecciones/documentos JSON/BSON).
  • Mongoose: ODM con modelos y validación.

Checklist de examen

  • Node no es lenguaje ni framework: es runtime.
  • Asíncrono/eventos: I/O no bloqueante (Event Loop).
  • API REST: rutas (GET/POST/PUT/DELETE), middlewares, JSON.
  • Mongo: documentos (JSON/BSON) en colecciones; ObjectId; índices si hay búsquedas.
  • Mongoose: esquema/modelo + validación.
Ejemplo completo: API Express + Mongoose (CRUD mínimo) 
// server.js
const express = require("express");
const mongoose = require("mongoose");

const app = express();
app.use(express.json());

mongoose.connect(process.env.MONGO_URI);

const UsuarioSchema = new mongoose.Schema({
  nombre: { type: String, required: true },
  email:  { type: String, required: true, unique: true },
}, { timestamps: true });

const Usuario = mongoose.model("Usuario", UsuarioSchema);

app.post("/usuarios", async (req, res) => {
  const u = await Usuario.create(req.body);
  res.status(201).json(u);
});

app.get("/usuarios", async (req, res) => {
  const lista = await Usuario.find().sort({ createdAt: -1 });
  res.json(lista);
});

app.listen(3000, () => console.log("API en :3000"));

Último repaso (2 minutos)

Si solo te da tiempo a una cosa: memoriza diferencias + comandos.

Diferencias que suelen caer

  • DNS: recursiva (me das respuesta final) vs iterativa (me rediriges).
  • Docker: imagen (plantilla) vs contenedor (instancia). Contenedor vs VM.
  • LDAP: directorio en árbol (lecturas/autenticación) vs MySQL (tablas/transacciones).
  • FTP vs SFTP: FTP no cifra; SFTP va sobre SSH.
  • Tomcat: contenedor de servlets/JSP; WAR en webapps.

Comandos-relámpago

  • DNS: nslookup (y si lo usáis: dig, host).
  • Docker: docker ps / images / logs / exec, docker compose up -d.
  • NGINX: nginx -t, systemctl reload nginx.
  • SFTP/SCP: sftp, scp.
